## Solution Code
```java
class Solution {
    public List<List<Integer>> shiftGrid(int[][] grid, int k) {

        int m = grid.length;
        int n = grid[0].length;

        int total = m * n;

        k %= total;

        List<List<Integer>> ans = new ArrayList<>();

        for (int i = 0; i < m; i++) {
            List<Integer> row = new ArrayList<>();

            for (int j = 0; j < n; j++) {
                row.add(0);
            }

            ans.add(row);
        }

        for (int i = 0; i < m; i++) {
            for (int j = 0; j < n; j++) {

                int oldIndex = i * n + j;

                int newIndex = (oldIndex + k) % total;

                int newRow = newIndex / n;
                int newCol = newIndex % n;

                ans.get(newRow).set(newCol, grid[i][j]);
            }
        }

        return ans;
    }
}
```

## Detailed Explanation
## 1️⃣ Algorithm Analysis – “What the code is doing”

The problem *Shift‑2d‑Grid* asks us to **rotate the entire grid as if it were a flat 1‑dimensional array of size `m·n`**.  
`k` shifts are performed: every element moves `k` positions forward (wrapping around the end).  

The accepted Java solution follows a very clean mathematical observation:

| Step | Observation |
|------|-------------|
| 1️⃣  | The grid can be seen as a sequence of `total = m·n` cells, numbered `0 … total‑1` in **row‑major order** (`i·n + j`). |
| 2️⃣  | A shift by `k` positions is equivalent to moving each cell with index `oldIdx` to<br>`newIdx = (oldIdx + k) % total`. |
| 3️⃣  | From `newIdx` we can recover the corresponding *output* coordinates: <br>`newRow = newIdx / n` (integer division) and `newCol = newIdx % n`. |
| 4️⃣  | Instead of physically moving every element many times (the naïve “repeat k swaps”), we **compute the final destination of each element once** and write it directly into a fresh answer matrix. |
| 5️⃣  | The answer matrix is pre‑allocated with the correct size; then a double loop visits every `(i,j)` in the original grid, computes where that element ends up after `k` shifts, and puts it there. |

So the algorithm is nothing more than **a single permutation of elements based on modular arithmetic**. It is *O(m·n)* time (we touch each element exactly once) and *O(m·n)* extra space for the result grid.

## 2️⃣ Step‑by‑Step Walkthrough (DRY‑Run)  

We will trace the sample input:

```
grid = [[1,2,3],
        [4,5,6],
        [7,8,9]]

k = 1
```

### Initial data

| Variable | Value |
|----------|-------|
| `m`      | 3 (rows) |
| `n`      | 3 (columns) |
| `total`  | `m * n = 9` |
| `k %= total` → `k = 1` (because 1 < 9) |

### Result matrix allocation  

`ans` is created with **3 rows**, each row is a list of **3 zeros**:

```
ans = [
        [0,0,0],
        [0,0,0],
        [0,0,0]
      ]
```

### Outer loops – iterating over every original cell

We go through each `(i, j)` in **row‑major order**.

| i | j | oldIndex = i·n + j | grid[i][j] | Explanation |
|---|---|-------------------|------------|-------------|
| 0 | 0 | `0·3+0 = 0` | `grid[0][0] = 1` | top‑left cell |
| 0 | 1 | `0·3+1 = 1` | `2` |
| 0 | 2 | `0·3+2 = 2` | `3` |
| 1 | 0 | `1·3+0 = 3` | `4` |
| 1 | 1 | `1·3+1 = 4` | `5` |
| 1 | 2 | `1·3+2 = 5` | `6` |
| 2 | 0 | `2·3+0 = 6` | `7` |
| 2 | 1 | `2·3+1 = 7` | `8` |
| 2 | 2 | `2·3+2 = 8` | `9` |

For each `oldIndex` we compute the **new index** after the rotation:

```
newIndex = (oldIndex + k) % total
        = (oldIndex + 1) % 9
```

| oldIndex | newIndex | newRow = newIndex / n | newCol = newIndex % n | Where we write `grid[i][j]` |
|----------|----------|----------------------|-----------------------|----------------------------|
| 0 | 1 | 1 / 3 = 0 | 1 % 3 = 1 | `ans[0][1] = 1` |
| 1 | 2 | 2 / 3 = 0 | 2 % 3 = 2 | `ans[0][2] = 2` |
| 2 | 3 | 3 / 3 = 1 | 3 % 3 = 0 | `ans[1][0] = 3` |
| 3 | 4 | 4 / 3 = 1 | 4 % 3 = 1 | `ans[1][1] = 4` |
| 4 | 5 | 5 / 3 = 1 | 5 % 3 = 2 | `ans[1][2] = 5` |
| 5 | 6 | 6 / 3 = 2 | 6 % 3 = 0 | `ans[2][0] = 6` |
| 6 | 7 | 7 / 3 = 2 | 7 % 3 = 1 | `ans[2][1] = 7` |
| 7 | 8 | 8 / 3 = 2 | 8 % 3 = 2 | `ans[2][2] = 8` |
| 8 | 0 | 0 / 3 = 0 | 0 % 3 = 0 | `ans[0][0] = 9` |

### Final `ans` matrix

Collecting the assignments:

```
ans = [
        [9, 1, 2],
        [3, 4, 5],
        [6, 7, 8]
      ]
```

That is exactly the expected output of shifting the grid **one step to the right** (or, equivalently, one step down when imagined as a linear array).  

Notice the subtle ordering: the *last* element of the original grid (`9`) wraps around to the very first cell (`ans[0][0]`). The traversal order ensures every element is placed correctly.

## 3️⃣ Complexity Analysis  

| Aspect | Derivation | Result |
|--------|------------|--------|
| **Time Complexity** | The algorithm contains two nested loops that iterate **exactly `m·n` times** (once per cell). Inside the loops we perform only O(1) arithmetic and a constant‑time `List#set`. | **O(m·n)** |
| **Auxiliary Space Complexity** | Apart from the input, we allocate:<br>• `ans`, an `m × n` matrix of integers → `m·n` entries.<br>• A few primitive variables (`i`, `j`, `oldIndex`, `newIndex`, etc.). | **O(m·n)** (the result itself) |
| **Overall Space** | Input grid (`m·n`) + result matrix (`m·n`) = `2·m·n` integers, which is still linear in the size of the input. | **O(m·n)** |

### Why it’s optimal  

* We must produce a **new grid** that contains all the original values, so at least `Ω(m·n)` space is unavoidable.  
* Each element has to be **read** and **written** at least once, giving a lower bound of `Ω(m·n)` operations. The algorithm meets both bounds exactly, making it asymptotically optimal.

## TL;DR Summary  

| What | How |
|------|-----|
| **Idea** | Treat the 2‑D grid as a flat array of length `m·n`. A shift by `k` moves each index `i` to `(i + k) % total`. Convert the new linear index back to `(row, col)` coordinates and write the value directly into the result matrix. |
| **Steps** | 1. Compute `total = m·n` and reduce `k` modulo `total`.<br>2. Allocate an empty `m×n` answer matrix.<br>3. For each original cell `(i,j)` compute `oldIdx = i·n + j`.<br>4. Compute `newIdx = (oldIdx + k) % total` and derive `newRow, newCol`.<br>5. Put `grid[i][j]` into `ans[newRow][newCol]`. |
| **Complexities** | **Time:** `O(m·n)` – one pass over all cells.<br>**Space:** `O(m·n)` – the output matrix (plus negligible overhead). |
| **Sample Run** | With `grid = [[1,2,3],[4,5,6],[7,8,9]]` and `k = 1`, the algorithm produces `[[9,1,2],[3,4,5],[6,7,8]]`. |

The solution is elegant, runs in linear time, and uses only the unavoidable amount of extra space for the result. It showcases a classic “modular permutation” trick that appears in many array‑rotation problems. 🎓
